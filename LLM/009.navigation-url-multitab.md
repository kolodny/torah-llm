## Boot-DB self-heal (deep links failed for stale OPFS)

A reported deep link errored because that browser's OPFS still held the iteration-1 boot DB
(no `links` table). Added `BOOT_VERSION` (`shared/schema.ts`): the slicer stamps it via
`PRAGMA user_version`, and the client re-downloads the boot DB if the stored version differs â€”
so an old cached DB self-heals instead of erroring.

## Multi-tab (the hard one)

A 2nd tab crashed with `NoModificationAllowedError: Access Handles cannot be createdâ€¦` â€” OPFS
**sync access handles are exclusive to one dedicated worker**. Things tried and why they failed:

- **SharedWorker holding the pool** â†’ `Missing required OPFS APIs` (sync access handles don't
  exist in a SharedWorker).
- **SharedWorker spawning a nested dedicated worker** (the canonical fix) â†’ bundles correctly for
  build, but **Vite dev doesn't transform a `new Worker`/`?worker` nested inside a worker**, so it
  hung in dev.

**Shipped:** each tab gets its **own dedicated worker + its own SAH pool**, named per-tab via a
`sessionStorage` id passed through the Worker `name` option (so a reload reuses the tab's pool).
Reliable in dev **and** build; tabs run simultaneously.

**Trade-off:** tabs don't share downloaded books (each re-downloads on demand â€” cheap over the
network for our small slices). A production-only enhancement could share one connection via a
SharedWorkerâ†’nested-worker broker.

Also: adding react-router mid-session left Vite's dep-optimize cache mixing two React copies
("Invalid hook call"); fixed by restarting the dev server with a cleared `.vite` cache.
