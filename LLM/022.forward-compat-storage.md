# 022 — Forward-compatible storage (incremental updates, never a full wipe)

**Date:** 2026-06-10

## Problem
`/db.sqlite` in OPFS was **both** the catalog (toc) and the accumulating content store (slices merged
in). A `BOOT_VERSION` mismatch ran `pool.wipeFiles()` — wiping *everything*. So any change (even a
one-word fix, or a schema tweak) forced a user who had downloaded multiple GB to **re-download the
whole corpus**. The goal: a DB change should only ever cost the **diff**, never a full re-download.

## Principle: schema vs. content are independent
| Concern | Mechanism | Cost of an update |
|---|---|---|
| **Schema** (table structure) | Migrate the cached `/db.sqlite` **in place** (`CONTENT_MIGRATIONS[v]` → v+1; additive `ALTER`/`CREATE`, or a table-rebuild). **Never** re-download to change schema. | ~0 (local SQL) |
| **Content** (rows: add / modify / delete books) | Per-book `content_version` (a row hash) + a global `publishId`. Only books whose hash changed re-download; removed books are pruned. | just the changed books |

`BOOT_VERSION` (in `shared/schema.ts`) is the schema version, stamped into the boot DB via
`PRAGMA user_version`. Every bump **must** add the matching `CONTENT_MIGRATIONS` step — a missing step
throws (a build bug) rather than silently wiping a multi-GB cache.

(User-authored data — notes, bookmarks — is **deferred**. It was the original motivation but isn't
built yet; when it lands it will live in its own never-wiped DB and reuse this same diff machinery.)

## Mechanism

### Publish side (`ingest/slice-db.ts`)
Computes `content_version` = short hash over each book's slice rows (editions + content + meta + links,
deterministic order) → `toc.content_version`. Emits `public/db/manifest.json`:
```json
{ "schemaVersion": 12, "publishId": "<hash over all (id, content_version)>", "books": <count> }
```
`publishId` changes iff any book changed. `BOOT_ONLY=1` regenerates just the boot DB + manifest
(reusing existing slices) when only versioning metadata changed.

### Local bookkeeping (worker, created locally — not shipped in slices)
- `book_state(toc_id PRIMARY KEY, content_version)` — the version actually merged locally.
- `cache_meta(key, value)` — stores the local `publishId`.

### Boot flow (`doBoot`)
1. No local `/db.sqlite` → download the boot DB fresh (ships at `user_version = BOOT_VERSION`).
2. Else open it and **migrate in place** (`CONTENT_MIGRATIONS`) up to `BOOT_VERSION`, each step in a
   transaction. Never resets/re-downloads. Then create the local bookkeeping tables.
3. Fetch `manifest.json`. Fresh download → just record `publishId`. Otherwise, if `publishId` differs:
   **refresh the catalog** (download the toc-only boot DB to a temp file, `INSERT OR REPLACE` toc,
   prune toc/content/editions/meta/links/book_state for removed books); on the first versioned boot,
   backfill `book_state` for already-cached books (trust the existing cache). Offline → skip, keep cache.

### Per-book diff application
- `ensureBook(tocId)` (re)merges a slice only when `book_state.content_version` is missing **or** `!=`
  `toc.content_version` — one check covers "never downloaded" and "stale".
- `mergeSlice()` is a **replace-merge** in a transaction: `DELETE` the book's editions/content/meta and
  its links (`from_id=? OR to_id=?` — a slice carries every link touching the book), `INSERT` from the
  slice, upsert `book_state`. content/links drop their surrogate ids (autoincrement locally) so a
  re-merge can't collide with another book's global ids; editions keep their stable TEXT id.

### Wipe
A single `wipe()` clears all local data — a **manual** recovery action (the header button), never an
automatic version wipe.

## Why this covers add / modify / delete / schema
- **Add a book** → new `toc` row (publishId changed → catalog refresh); downloaded on demand.
- **Modify a book** → its `content_version` changes → replace-merge on next open. Verified: a sentinel
  row added to Exodus (3630→3631, replace = no dup) while Genesis was untouched.
- **Delete a book** → removed from the catalog → reconcile prunes its toc/content/editions/links/
  book_state. Verified: deleting Exodus pruned it fully; Genesis intact.
- **Schema change** → in-place migration; no re-download. Verified: a real prior v11 OPFS migrated
  11→12 in place (added `content_version`) preserving the already-downloaded book.

## Decisions still open (see the conversation)
- **Migration libraries** — not used. The in-place ladder is ~20 lines and the project is raw-SQL by
  design; Drizzle/Kysely/Umzug would add deps + abstraction and still need adapting to
  sqlite-wasm-in-a-worker. Schema migrations don't need a library.
- **Sub-book "version patches"** — today a changed book re-downloads its *whole slice* (already far
  better than the corpus). SQLite's **session extension is in our build** and **`sqldiff --changeset`**
  generates a row-level changeset between two DBs, so true deltas are feasible. Caveat: changesets key
  on PRIMARY KEY, but the client autoincrements content/links ids locally — using changesets would need
  PK-stability rework + the build retaining the prior publish + version chaining (or full-slice
  fallback). Pending a decision on whether the bandwidth win is worth that complexity now.
- **User data** (notes/bookmarks) — deferred; own never-wiped DB, same diff machinery, when wanted.
