# 050 — Result links get real hrefs (cmd-click) + segment.tsx strip drops mesorah markers

**Date:** 2026-06-28

## Why

- Plugin result links (pasuk-name, gematria, notes, search) were `<Anchor onClick={navigate}>` with **no
  `href`** — so cmd/ctrl/middle-click couldn't open a verse in a new tab, and the URL didn't show on hover. The
  Code page already did this right (`linkTo` builds a real `href`); the plugins should match.
- `src/workbench/segment.tsx` kept its **own** tag-only `stripTags` (separate from the [049](049.strip-mesorah-markers.md)
  shared fix), so a segment's `text` (exposed to plugins, and the basis for decoration offsets) still contained
  the `{פ}`/`{ס}` parsha markers.

## What shipped

- **`ctx.ui.href(book, ref?)`** (types + host) returns the in-app verse URL
  (`?page=viewer&book=…&ref=…`, relative so it works under any base path). The four plugin link Anchors now set
  `href={ctx.ui.href(...)}` and a modifier-aware `onClick` — plain left-click `preventDefault`s and SPA-navigates;
  cmd/ctrl/shift/middle-click falls through to open a new tab. (songs uses `peek`, a preview, left as-is.)
- **`segment.tsx` strip is now marker-consistent.** Its `stripTags` drops `mam-spi` spans (via the shared
  `MAM_SPI_SPAN`, now exported), and `decorateHtml`'s char-walk treats a `mam-spi` span as **zero-width** (kept
  verbatim in the rendered HTML, contributing no plaintext). Crucially it stays **tag-only / entity-raw** (it does
  *not* use the shared `stripHtml`, which decodes entities — that would desync mark offsets against the raw walk).
  So `seg.text` is clean for plugin analysis while offsets stay aligned with what's rendered.

## Verified

- In-browser: pasuk-name links render `href="?page=viewer&book=Genesis&ref=7%3A8"` (cmd-click opens a new tab);
  Code-page auto-links likewise.
- Node check: `stripTags` length == `decorateHtml` plaintext walk length for a petucha span, a setuma span, an
  entity (`&thinsp;`), and a ketiv/qere span (content preserved). `npx tsc` clean.
