# 035 — Cantillation SQL sample (code page example, iterated)

**Date:** 2026-06-22

## Why

The Monaco **SQL Code page** ([028](028.code-page-and-plugin-api.md)) ships sample queries that double
as documentation for its `evalJS` capability. This adds a genuinely hard one — *find a Torah word that
bears two distinct cantillation marks (te'amim)* — and iterates it until it returns only the truly
notable cases. It exercises `evalJS` over Hebrew Unicode and is a good stress test of the sample API;
it is **not** core behavior (no cantillation logic was special-cased into the worker — per the user's
"it shouldn't be special-cased here" correction, it lives entirely as an `evalJS` expression in the sample).

## What shipped (`plugins/code-search/CodePage.tsx`)

Iterated over four commits as spot-checking across the Chumash exposed false positives:

1. **`785c228`** — initial: scan the U+0591–U+05AF range, collect accents into a `Set` so the same
   accent doubled for layout (pashta/telisha) doesn't double-count.
2. **`6dd2187`** — require two **distinct disjunctive** te'amim: drop conjunctives (U+05A3–U+05AA), so
   a servant+disjunctive pair (munaḥ+zaqef, qadma+geresh) no longer counts; merge zinor→zarqa
   (one accent, two positional codepoints); also split on maqaf (־).
3. **`290bcba`** — tokenize on **paseq (׀)** and HTML spacing entities (`&nbsp;`/`&thinsp;`): poetic
   layout (Az Yashir, Ha'azinu) was joining two words into one token and matching them. After this it
   cleanly surfaces the real phenomena — the Ten Commandments' dual **ta'am elyon/taḥton** (Ex 20,
   Deut 5) and the **Reuben pisḳa** (Gen 35:22).
4. **`f5c0f03`** — tidy only: reformat the one-line `evalJS` blob into a readable IIFE with named
   constants and hex codepoints; a single `evalJS` in a subquery rather than duplicated in `WHERE`.
   Same results.

## Takeaway

Hebrew cantillation has several ways to *look* like "two marks" that aren't (conjunctive+disjunctive
pairs, layout-doubled glyphs, multi-codepoint single accents, words merged by spacing/paseq). The
working filter is: tokenize on whitespace + maqaf + paseq + spacing entities, keep only **disjunctive**
accents, normalize zinor→zarqa, and require **≥2 distinct** per token.

## Followup (2026-06-23) — replaced the evalJS with pure SQL

The `evalJS` IIFE was hard to read. Rewrote the sample as **pure SQL, no evalJS and no new functions**: a
recursive CTE splits each Hebrew verse into words (on space / `MAQAF()` / `PASEQ()`), a second recursion
**explodes each word into characters**, and a `COUNT(DISTINCT CASE WHEN ch IN (ETNAHTA(), SEGOLTA(), …,
ZINOR()) THEN … END)` counts distinct **disjunctive** te'amim per word — the disjunctive accents named in the
`IN(...)` (conjunctive "servants" omitted), `zinor` folded into `zarqa`; `>= 2`, grouped to one row per
verse. (Two earlier cuts: a 21-term `(instr(word, <MARK>()) > 0)` sum — ugly; then a `cp BETWEEN 0x0591 AND
0x05AE …` codepoint range — terse but cryptic. The named `IN(...)` set reads best and the marks are
`deterministic` 0-arity functions so SQLite constant-folds the list — no per-char cost.) Scoped to
Genesis/Exodus/Deuteronomy it returns the same 19 famous verses (the Decalogue in Exodus 20 + Deut 5, plus
Genesis 5:29 and the Reuben pisḳa 35:22) in ~0.9 s — a nice showcase of SQLite recursive CTEs.

Also fixed `readOnlyQuery` (the plugin SELECT-only guard): it rejected any non-trailing `;`, but checked the
raw text, so a `;` inside a comment or string literal was a false positive (this very sample's comment
tripped it). It now blanks out comments + string/identifier literals before the multi-statement check.

## Resolution (2026-06-23, [040](040.code-samples-polish.md))

Settled on a **far simpler signal** than any of the above. All te'amim occupy one Unicode block
(U+0591–U+05AE), and a doubly-accented letter is just **two of those marks adjacent** in the string — so
the entire filter is one regex, `/[֑-֮]{2}/.test(value)`, via evalJS. No recursive CTE, no
disjunctive enumeration, no zinor fold (the earlier evalJS needed the fold only because it counted
*distinct disjunctives across a whole word*; the positional "two-in-a-row" test sidesteps all of it —
Deut 12:18's zarqa+zinor sit on different letters, never adjacent). **21 verses, ~34 ms**: the full
Decalogue dual ta'am elyon/taḥton (incl. the short prohibitions the distinct-set method dropped) + Gen
35:22 + Gen 5:29 — cleaner *and* more correct than the 19-verse versions. See [040](040.code-samples-polish.md).
