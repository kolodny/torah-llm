# 040 — Code-page sample queries polish (all examples verified)

**Date:** 2026-06-23

## Why

The Monaco **SQL Code page** ([028](028.code-page-and-plugin-api.md)) ships sample queries that double as
documentation. They had accreted over [029](029.gematria-torah-code.md)/[035](035.cantillation-sample.md);
the user wanted every example clean, correct, and genuinely interesting — and singled out the cantillation
sample as "pretty bad" (the verbose 30-line recursive CTE from 035's followup). This is a QA/polish pass
that judged each example, rewrote the weak ones, and **verified every query against the live dev DB**
(throwaway Playwright over `window.__torahDb.sql`, books downloaded via `ensureBook`).

## What shipped

**Cantillation — "A word with two cantillation marks (Torah)"** (`plugins/code-search/CodePage.tsx`):
went through several false starts — 035's verbose pure-SQL recursive CTE, then a "distinct-disjunctive-set
+ zinor→zarqa fold" evalJS — both of which over-thought it. The **right** signal is far simpler: every
ta'am lives in **one Unicode block (U+0591–U+05AE)**, and a doubly-accented letter is just **two of those
marks in a row**. So the whole test is one tiny regex — `/[֑-֮]{2}/.test(value)` — with no
disjunctive enumeration, no zinor fold, no word-splitting. One `evalJS` in `WHERE`, one in `SELECT` (to
pull the marked word for display via `strip(c.text)` → `value.split(/\s+/).find(...)`). ~34 ms.
**Result: 21 verses** — the *full* Decalogue dual ta'am elyon/taḥton (Ex 20:2–13, Deut 5:6–17, incl. the
short prohibitions לֹא תרצח… that the distinct-set method *missed*) plus **Gen 35:22** (Reuben pisḳa,
`רְאוּבֵ֗֔ן`) and **Gen 5:29** (Noaḥ, `זֶ֞֠ה`). Deut 12:18 (the old zinor false positive) is correctly
excluded — its zarqa and zinor sit on different letters, never adjacent. Cleaner *and* more correct than
the 19-verse versions.

**Pazer sample** — broadened from Genesis-only (1 row) to the whole Chumash (7 rows); reworded to
explain pazer is a rare disjunctive and the `length - length(replace())` counting idiom. Kept.

**Gematria** (`plugins/gematria/index.tsx`): **dropped `gematria:prime`** — it returned a single row over
Genesis 1 and was redundant with `gematria:triangular` (both "evalJS number-theory test on a verse
gematria"). The remaining six (verse value, equal-gematria pairs, verse↔Rashi, verse-equals-word,
triangular 2701 = 73rd, words=376) all verified correct and kept. (`gematria()` is a SQL fn and `H` is the
char-*name* map, so the recursive-CTE word-split that reuses `gematria(word)` is the right tool — not
replaceable by a clean evalJS.)

**Torah-code** (`plugins/torah-code/index.tsx`): both samples verified and **kept as-is** — תורה found at
exactly skip 50 in Genesis (the classic result), אברהם at skip 13. Already clean.

Other CodePage samples (browse, cross-reference graph → Gen 1:1 = 2230 links, most-repeated → 69×
Vayedaber, Gen 1:1 = 7 words/28 letters, longest chapter → Numbers 7 = 89, largest books) all verified
correct and kept unchanged.

## Followup (2026-06-23) — redundancy sweep + a cross-book cantillation sample

A second pass on the user's prompt to kill "silly mistakes":

- **Alias-in-WHERE.** The cantillation sample ran its `evalJS` regex twice per row (once in `SELECT … AS
  word`, once in `WHERE`); the pazer sample likewise repeated its `length(text) - length(replace(…))`
  expression. SQLite lets `WHERE` reference an output alias, so both now filter on `WHERE word IS NOT NULL`
  / `WHERE pazer_count >= 2` — half the per-row work, and the intent reads straight off the alias.
- **Compute-once.** `gematria:triangular` called `gematria(c.text)` twice (display + the evalJS arg); wrapped
  the verse in a subquery so the gematria is computed once and both columns read the alias.
- **Verified every comment claim live**, not just that queries run: Gen 1:1 = 7 words/28 letters; Numbers 7 =
  89 verses; 2701 = 73rd triangular; most-referenced = Genesis 1:1 (2230); most-repeated = 69× Vayedaber +
  12× the princes; words = 376 → שלום/עשו/וישכם; verse-equals-word's "Shema (Deut 6:4)" → 1118 = תשחית and
  "Exodus 15:18 = 376". All held.
- **New sample — "Which book has the most double-cantillation verses?"** (`GROUP BY toc_id`, corpus-wide
  over downloaded books). Verifying it overturned my own first-draft comment: the prose Decalogue does **not**
  lead — **Psalms (161), Proverbs (51), Job** dominate, because the three poetic *sifrei emet* books use a
  separate cantillation system that stacks accents far more often. Comment rewritten to teach that, ~60 ms.

The lesson from the original pass repeats: don't trust a claim (even your own) until the DB confirms it.

- **Results table → TanStack Table (sortable), and a layout fix.** The hand-rolled table (1) capped height
  at 600 px in a nested `ScrollArea` inside the already-scrolling `.plugin-page`, so a 25-row page was
  clipped — far worse on mobile — and (2) wasn't sortable. Replaced the `ScrollArea` with
  `Table.ScrollContainer` (horizontal-only; the page handles vertical scroll, so every row renders), and
  rebuilt sorting/pagination on **`@tanstack/react-table`** (headless) while keeping Mantine's `<Table>` and
  our `renderCell()` (link/render/auto-link cells). Click a header to sort; a `sortValue()` accessor makes
  numbers sort numeric, `link()` cells sort by ref, `render()` matrix cells sort last. Verified 25 rows +
  working sort (refs sort 1:31 > 1:30 > … naturally) on desktop and mobile.
- **`verse-equals-word` → one row per (verse, word) pair.** It first showed `min(w)` (the alphabetically-first
  word, e.g. אֶעֱשֶׂה for 376) — one example per verse, which *hid* recognizable matches like שלום. Per the user,
  the right model is **distinct over (verse, word)**: a verse whose total matches N distinct words gets N rows.
  Replaced the single-example `wval` with a `matches` CTE of every distinct word (by consonantal form) + its
  gematria, joined to the distinct-verse set on gematria. Exodus 15:18 (376) now lists all 27 matching words
  incl. עֵשָׂו **and** שָׁלוֹם; recurring verses still collapse (occurrences count); 754 rows, 0 duplicate pairs.
  (The sibling "376 = שלום = עשו" words sample already lists שלום, ×11.)

- **Collapsed duplicate rows in the two gematria word samples.** `verse-equals-word` keyed on the gematria
  value, so a verse that recurs verbatim (e.g. וַיְדַבֵּר יְהֹוָה אֶל מֹשֶׁה לֵּאמֹר, ~69×) produced ~69 identical
  rows. Now `GROUP BY c.text` collapses identical verses to one (bare `toc_id`/`ref` give a valid
  representative occurrence, `count(*) AS occurrences` shows the multiplicity) — 204 → 132 rows, and a
  genuinely *different* verse with the same gematria stays its own row. `words-equal` likewise listed every
  occurrence (עשו ×118); now `GROUP BY` the consonants-only form (`evalJS('value.replace(/[^֐-׮... ]/,…)')`,
  since te'amim vary per occurrence) yields 27 distinct words with occurrence counts. Audited the rest:
  no other sample dups, because each Torah book has exactly **one** Hebrew Sefaria edition (verified), so
  the `content × editions` join never multiplies — gematria:verse (31, 0 dup), gematria:pairs (224, 0 dup),
  torah-code's `group_concat` stream (1 clean copy) are all fine as-is.

- **Killed the recursive word-split CTE.** The two gematria word samples (`words-equal`, `verse-equals-word`)
  split verses into words with a verbose `WITH RECURSIVE src … words(…)` boilerplate so SQL's `gematria()`
  could score each. `evalJS` can't call `gematria()` (its scope is only `value/args/strip/H` — `H` is the
  char-*name* map, not letter values), so the split had to stay in SQL. New idiom: **`evalJS` does just the
  split — returns the words as a JSON array — and `json_each(evalJS(…))` unnests them**, leaving `gematria()`
  in SQL. The whole recursive CTE collapses to one `JOIN json_each(...)`. Verified byte-identical results
  (205 / 204 rows, same famous hits) at the same speed.

- **Dropped every presentation `LIMIT`.** The Code page materializes the whole result set in one
  `ctx.data.query()` and `ResultsTable` paginates client-side (`rows.slice`), so a `LIMIT` only truncated
  what you could page to — it never reduced query cost (incremental fetch isn't possible: it would mean
  re-running the query per page with `OFFSET`, which fights the connection lease and re-scans). Measured the
  unbounded versions first: every sample is bounded by its own `WHERE`/scope, the largest (most-referenced)
  tops out at ~5,846 rows / 481 ms because it's filtered to the five Chumashim — a ceiling that doesn't grow
  as you download more books. All well under the 10 s timeout, so the caps were pure noise. Kept `LIMIT` only
  as an editor autocomplete keyword.

---

We spent three iterations modelling cantillation as a *set-membership* problem (which disjunctives? fold
zinor?) when the phenomenon is *positional*: two accent marks adjacent in the string. Once framed that
way the filter collapses to a 9-character regex over the U+0591–U+05AE block — cleaner than both the
recursive CTE and the distinct-set evalJS, faster (34 ms), and more correct (21 verses: it no longer
drops the Decalogue's short prohibitions). Lesson: before enumerating a domain set, check whether the
feature has a simpler structural signature. `npx tsc --noEmit` clean.
