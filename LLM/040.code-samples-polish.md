# 040 — Code-page sample queries polish (all examples verified)

**Date:** 2026-06-23

## Why

The Monaco **SQL Code page** ([028](028.code-page-and-plugin-api.md)) ships sample queries that double as
documentation. They had accreted over [029](029.gematria-torah-code.md)/[035](035.cantillation-sample.md);
the user wanted every example clean, correct, and genuinely interesting — and singled out the cantillation
sample as "pretty bad" (the verbose 30-line recursive CTE from 035's followup). This is a QA/polish pass
that judged each example, rewrote the weak ones, and **verified every query against the live dev DB**
(throwaway Playwright over `window.__torahDb.sql`, books downloaded via `ensureBook`).

## What shipped

**Cantillation — "A word with two cantillation marks (Torah)"** (`plugins/code-search/CodePage.tsx`):
went through several false starts — 035's verbose pure-SQL recursive CTE, then a "distinct-disjunctive-set
+ zinor→zarqa fold" evalJS — both of which over-thought it. The **right** signal is far simpler: every
ta'am lives in **one Unicode block (U+0591–U+05AE)**, and a doubly-accented letter is just **two of those
marks in a row**. So the whole test is one tiny regex — `/[֑-֮]{2}/.test(value)` — with no
disjunctive enumeration, no zinor fold, no word-splitting. One `evalJS` in `WHERE`, one in `SELECT` (to
pull the marked word for display via `strip(c.text)` → `value.split(/\s+/).find(...)`). ~34 ms.
**Result: 21 verses** — the *full* Decalogue dual ta'am elyon/taḥton (Ex 20:2–13, Deut 5:6–17, incl. the
short prohibitions לֹא תרצח… that the distinct-set method *missed*) plus **Gen 35:22** (Reuben pisḳa,
`רְאוּבֵ֗֔ן`) and **Gen 5:29** (Noaḥ, `זֶ֞֠ה`). Deut 12:18 (the old zinor false positive) is correctly
excluded — its zarqa and zinor sit on different letters, never adjacent. Cleaner *and* more correct than
the 19-verse versions.

**Pazer sample** — broadened from Genesis-only (1 row) to the whole Chumash (7 rows); reworded to
explain pazer is a rare disjunctive and the `length - length(replace())` counting idiom. Kept.

**Gematria** (`plugins/gematria/index.tsx`): **dropped `gematria:prime`** — it returned a single row over
Genesis 1 and was redundant with `gematria:triangular` (both "evalJS number-theory test on a verse
gematria"). The remaining six (verse value, equal-gematria pairs, verse↔Rashi, verse-equals-word,
triangular 2701 = 73rd, words=376) all verified correct and kept. (`gematria()` is a SQL fn and `H` is the
char-*name* map, so the recursive-CTE word-split that reuses `gematria(word)` is the right tool — not
replaceable by a clean evalJS.)

**Torah-code** (`plugins/torah-code/index.tsx`): both samples verified and **kept as-is** — תורה found at
exactly skip 50 in Genesis (the classic result), אברהם at skip 13. Already clean.

Other CodePage samples (browse, cross-reference graph → Gen 1:1 = 2230 links, most-repeated → 69×
Vayedaber, Gen 1:1 = 7 words/28 letters, longest chapter → Numbers 7 = 89, largest books) all verified
correct and kept unchanged.

## Takeaway

We spent three iterations modelling cantillation as a *set-membership* problem (which disjunctives? fold
zinor?) when the phenomenon is *positional*: two accent marks adjacent in the string. Once framed that
way the filter collapses to a 9-character regex over the U+0591–U+05AE block — cleaner than both the
recursive CTE and the distinct-set evalJS, faster (34 ms), and more correct (21 verses: it no longer
drops the Decalogue's short prohibitions). Lesson: before enumerating a domain set, check whether the
feature has a simpler structural signature. `npx tsc --noEmit` clean.
