## What shipped

**Cantillation вЂ” "A word with two cantillation marks (Torah)"** (`plugins/code-search/CodePage.tsx`):
went through several false starts вЂ” 035's verbose pure-SQL recursive CTE, then a "distinct-disjunctive-set
+ zinorв†’zarqa fold" evalJS вЂ” both of which over-thought it. The **right** signal is far simpler: every
ta'am lives in **one Unicode block (U+0591вЂ“U+05AE)**, and a doubly-accented letter is just **two of those
marks in a row**. So the whole test is one tiny regex вЂ” `/[Ц‘-Ц®]{2}/.test(value)` вЂ” with no
disjunctive enumeration, no zinor fold, no word-splitting. One `evalJS` in `WHERE`, one in `SELECT` (to
pull the marked word for display via `strip(c.text)` в†’ `value.split(/\s+/).find(...)`). ~34 ms.
**Result: 21 verses** вЂ” the *full* Decalogue dual ta'am elyon/taбёҐton (Ex 20:2вЂ“13, Deut 5:6вЂ“17, incl. the
short prohibitions ЧњЦ№Чђ ЧЄЧЁЧ¦Ч—вЂ¦ that the distinct-set method *missed*) plus **Gen 35:22** (Reuben pisбёіa,
`ЧЁЦ°ЧђЧ•ЦјЧ‘ЦµЦ—Ц”Чџ`) and **Gen 5:29** (NoaбёҐ, `Ч–Ц¶ЦћЦ Ч”`). Deut 12:18 (the old zinor false positive) is correctly
excluded вЂ” its zarqa and zinor sit on different letters, never adjacent. Cleaner *and* more correct than
the 19-verse versions.

**Pazer sample** вЂ” broadened from Genesis-only (1 row) to the whole Chumash (7 rows); reworded to

**Gematria** (`plugins/gematria/index.tsx`): **dropped `gematria:prime`** вЂ” it returned a single row over

## Followup (2026-06-23) вЂ” redundancy sweep + a cross-book cantillation sample

A second pass on the user's prompt to kill "silly mistakes":

- **Alias-in-WHERE.** The cantillation sample ran its `evalJS` regex twice per row (once in `SELECT вЂ¦ AS
  word`, once in `WHERE`); the pazer sample likewise repeated its `length(text) - length(replace(вЂ¦))`
  expression. SQLite lets `WHERE` reference an output alias, so both now filter on `WHERE word IS NOT NULL`
  / `WHERE pazer_count >= 2` вЂ” half the per-row work, and the intent reads straight off the alias.
- **Compute-once.** `gematria:triangular` called `gematria(c.text)` twice (display + the evalJS arg); wrapped
  the verse in a subquery so the gematria is computed once and both columns read the alias.
- **Verified every comment claim live**, not just that queries run: Gen 1:1 = 7 words/28 letters; Numbers 7 =
  89 verses; 2701 = 73rd triangular; most-referenced = Genesis 1:1 (2230); most-repeated = 69Г— Vayedaber +
  12Г— the princes; words = 376 в†’ Ч©ЧњЧ•Чќ/ЧўЧ©Ч•/Ч•Ч™Ч©Ч›Чќ; verse-equals-word's "Shema (Deut 6:4)" в†’ 1118 = ЧЄЧ©Ч—Ч™ЧЄ and
  "Exodus 15:18 = 376". All held.
- **New sample вЂ” "Which book has the most double-cantillation verses?"** (`GROUP BY toc_id`, corpus-wide
  over downloaded books). Verifying it overturned my own first-draft comment: the prose Decalogue does **not**
  lead вЂ” **Psalms (161), Proverbs (51), Job** dominate, because the three poetic *sifrei emet* books use a
  separate cantillation system that stacks accents far more often. Comment rewritten to teach that, ~60 ms.

The lesson from the original pass repeats: don't trust a claim (even your own) until the DB confirms it.

- **Two reusable built-ins replace repeated JS in the samples** (`src/db/worker.ts`, alongside `strip`):
  `words(text)` в†’ a JSON array of a verse's words (HTML-stripped, split on whitespace + maqaf + paseq), pair
  with `json_each()`; `letters(text)` в†’ only the Hebrew letters U+05D0вЂ“U+05EA (strips nikud/te'amim/punct/HTML),
  a consonantal key or `length(letters(x))` letter count. They retired the `evalJS('JSON.stringify(strip(value)
  .split(вЂ¦))')` tokeniser and the `evalJS('value.replace(/[^Цђ-Ч®]/,вЂ¦)')` consonant key in the gematria samples,
  and turned **Genesis 1:1** into pure SQL (`json_array_length(words(c.text))` = 7, `length(letters(c.text))` =
  28 вЂ” no evalJS). Both are in autocomplete + the built-ins help line.
- **`verse-equals-word` final shape:** dropped the `occurrences` column; matching words `group_concat(вЂ¦, ' / ')`
  into one cell per verse; and the per-cell 200-char truncation in the results table is gone (cells just wrap
  and grow taller, so the full joined list shows).
- **Editor state in the URL.** CodePage mirrors the SQL + selected sample into `?sql=&sample=` via
  `history.replaceState` (not push вЂ” typing mustn't pile up Back entries), preserving other params; on mount it
  seeds state from the URL, so a reload restores the editor. Verified: reload в†’ same query runs.
- **A third helper + two new samples.** Added `chapter(ref)` (the chapter number of a `chapter:verse` ref,
  or NULL) and used it to drop the `CAST(substr(ref, 1, instr(ref,':')-1))` idiom from the longest-chapters
  sample. New samples showcasing the helpers: **"The Torah's most-frequent words"** (`words()`+`letters()`
  в†’ ЧђЦµЧЄ Г—2622, ЧђЦІЧ©Ц¶ЧЃЧЁ Г—1664, the Divine Name Г—1526) and **"The longest verses (by word count)"**
  (`json_array_length(words(c.text))` в†’ Exodus 32:1 at 34 вЂ” the clean redo of the old space-counting
  "longest verse" that poetic layout had broken). All three helpers are in autocomplete + the help line.
- **Spell-check:** added the domain terms to `cspell.json` and an `ignoreRegExpList` rule for the Hebrew
  Unicode block (`/[\\u0590-\\u05FF\\uFB1D-\\uFB4F]+/g`) so Hebrew text in comments isn't flagged word by
  word (the v10 key is `ignoreRegExpList`, not `ignoreRegExp`).

- **Results table в†’ TanStack Table (sortable), and a layout fix.** The hand-rolled table (1) capped height
  at 600 px in a nested `ScrollArea` inside the already-scrolling `.plugin-page`, so a 25-row page was
  clipped вЂ” far worse on mobile вЂ” and (2) wasn't sortable. Replaced the `ScrollArea` with
  `Table.ScrollContainer` (horizontal-only; the page handles vertical scroll, so every row renders), and
  rebuilt sorting/pagination on **`@tanstack/react-table`** (headless) while keeping Mantine's `<Table>` and
  our `renderCell()` (link/render/auto-link cells). Click a header to sort; a `sortValue()` accessor makes
  numbers sort numeric, `link()` cells sort by ref, `render()` matrix cells sort last. Verified 25 rows +
  working sort (refs sort 1:31 > 1:30 > вЂ¦ naturally) on desktop and mobile.
- **`verse-equals-word` в†’ one row per (verse, word) pair.** It first showed `min(w)` (the alphabetically-first
  word, e.g. ЧђЦ¶ЧўЦ±Ч©Ц¶Ч‚Ч” for 376) вЂ” one example per verse, which *hid* recognizable matches like Ч©ЧњЧ•Чќ. Per the user,
  the right model is one row per verse listing **all** its matching words. Replaced the single-example `wval`
  with a `matches` CTE of every distinct word (by consonantal form) + its gematria, joined to the
  distinct-verse set on gematria; `group_concat(word, ' / ')` then folds the matches into one cell per verse
  (`word_count` shows how many). Exodus 15:18 (376) now reads `вЂ¦ / ЧўЦµЧ©ЦёЧ‚Ч• / вЂ¦ / Ч©ЦёЧЃЧњЧ•Ц№Чќ` (27 words); recurring
  verses still collapse (occurrences); 132 rows. (An earlier cut emitted one row per (verse, word) = 754 rows вЂ”
  too many; the user asked to join the words per verse instead.) (The sibling words sample already lists Ч©ЧњЧ•Чќ.)

- **Collapsed duplicate rows in the two gematria word samples.** `verse-equals-word` keyed on the gematria
  value, so a verse that recurs verbatim (e.g. Ч•Ц·Ч™Ц°Ч“Ц·Ч‘ЦµЦјЧЁ Ч™Ц°Ч”Ц№Ч•ЦёЧ” ЧђЦ¶Чњ ЧћЦ№Ч©Ц¶ЧЃЧ” ЧњЦµЦјЧђЧћЦ№ЧЁ, ~69Г—) produced ~69 identical
  rows. Now `GROUP BY c.text` collapses identical verses to one (bare `toc_id`/`ref` give a valid
  representative occurrence, `count(*) AS occurrences` shows the multiplicity) вЂ” 204 в†’ 132 rows, and a
  genuinely *different* verse with the same gematria stays its own row. `words-equal` likewise listed every
  occurrence (ЧўЧ©Ч• Г—118); now `GROUP BY` the consonants-only form (`evalJS('value.replace(/[^Цђ-Ч®... ]/,вЂ¦)')`,
  since te'amim vary per occurrence) yields 27 distinct words with occurrence counts. Audited the rest:
  no other sample dups, because each Torah book has exactly **one** Hebrew Sefaria edition (verified), so
  the `content Г— editions` join never multiplies вЂ” gematria:verse (31, 0 dup), gematria:pairs (224, 0 dup),
  torah-code's `group_concat` stream (1 clean copy) are all fine as-is.

- **Killed the recursive word-split CTE.** The two gematria word samples (`words-equal`, `verse-equals-word`)
  split verses into words with a verbose `WITH RECURSIVE src вЂ¦ words(вЂ¦)` boilerplate so SQL's `gematria()`
  could score each. `evalJS` can't call `gematria()` (its scope is only `value/args/strip/H` вЂ” `H` is the
  char-*name* map, not letter values), so the split had to stay in SQL. New idiom: **`evalJS` does just the
  split вЂ” returns the words as a JSON array вЂ” and `json_each(evalJS(вЂ¦))` unnests them**, leaving `gematria()`
  in SQL. The whole recursive CTE collapses to one `JOIN json_each(...)`. Verified byte-identical results
  (205 / 204 rows, same famous hits) at the same speed.

- **Dropped every presentation `LIMIT`.** The Code page materializes the whole result set in one
  `ctx.data.query()` and `ResultsTable` paginates client-side (`rows.slice`), so a `LIMIT` only truncated
  what you could page to вЂ” it never reduced query cost (incremental fetch isn't possible: it would mean
  re-running the query per page with `OFFSET`, which fights the connection lease and re-scans). Measured the
  unbounded versions first: every sample is bounded by its own `WHERE`/scope, the largest (most-referenced)
  tops out at ~5,846 rows / 481 ms because it's filtered to the five Chumashim вЂ” a ceiling that doesn't grow
  as you download more books. All well under the 10 s timeout, so the caps were pure noise. Kept `LIMIT` only
  as an editor autocomplete keyword.

---

We spent three iterations modelling cantillation as a *set-membership* problem (which disjunctives? fold
zinor?) when the phenomenon is *positional*: two accent marks adjacent in the string. Once framed that
way the filter collapses to a 9-character regex over the U+0591вЂ“U+05AE block вЂ” cleaner than both the
recursive CTE and the distinct-set evalJS, faster (34 ms), and more correct (21 verses: it no longer
drops the Decalogue's short prohibitions). Lesson: before enumerating a domain set, check whether the
feature has a simpler structural signature. `npx tsc --noEmit` clean.
