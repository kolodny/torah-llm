# 030 — Storage tree, tags, gzipped slices, build split

**Date:** 2026-06-11

## Why
The Storage page needed a precise, reusable book selector (also wanted by the Gematria page); users wanted
quick group-select ("Torah", "Talmud", …); and per-book transfers were larger than necessary.

## What shipped
- **`BookCheckTree`** (`src/components/BookTree.tsx`) — a reusable nested catalog checkbox tree with
  cascading select + **indeterminate** parents; pre-checks already-downloaded books. Used by both the
  Storage page and the Gematria search page.
- **Tags** — quick-select chips above the tree: the curated **Tanakh divisions** (Torah / Nevi'im / Ketuvim
  / Tanakh, base-text-only lists, since Sefaria's "Tanakh" mixes in commentaries) **plus every other
  top-level Sefaria category auto-extracted** (Talmud, Mishnah, Midrash, …). Clicking a chip toggles all its
  books. Empty tags hidden; counts shown.
- **Storage page** — diff model: `toDownload = checked − local`, `toClear = local − checked`; the Download
  button shows the total size to fetch.
- **Gzipped slices** — the slicer gzips each per-book slice and the boot DB (`.gz`); the worker prefers
  `.gz` and inflates client-side via `DecompressionStream('gzip')` (skips inflate if the server already set
  `Content-Encoding: gzip`). ~5× smaller transfer. (Later made catalog-sensitive + index-free — see
  [031](031.catalog-placement.md), [032](032.github-pages-publish.md).)
- **Build split** — `npm run build` builds only the app (~6 s; `copyPublicDir:false` + `emptyOutDir:false`
  so it never churns the multi-GB `public/db`), while `npm run build:db` rsyncs the corpus into `dist/db`
  separately.
