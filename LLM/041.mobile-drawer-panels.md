# 041 — Mobile panels drawer rendered empty (peek/notes/search blank)

**Date:** 2026-06-24

## Why

On mobile, tapping a cross-reference/commentary link (or opening the panels FAB) showed the right-rail
**Drawer with its tabs — Preview / Notes / Search / Songs — but a completely blank body**. It looked like
"a sidebar with a bunch of tabs that are all empty." Desktop was fine. Reproduced on the deployed app under
iPhone emulation; the `.peek-body` was present in the DOM with content (height ~126 px) but **clipped to
invisibility**.

## Root cause

The right rail (`ViewerSidebar`) is a flex column: a `Tabs` bar + a `.viewer-side-body` that's `flex: 1 1 0;
overflow: auto`. On desktop `.viewer-side` sits in the `.panes` flex row (`align-items: stretch`) so it gets
full height. On mobile it's hosted inside a **Mantine `Drawer`**, whose **content panel is not a flex
column**, so the drawer body (`.aside-drawer-body`) collapsed to the tab-bar height (~36 px). With the body
at 36 px, `.viewer-side`'s `height: 100%` resolved against an effectively auto-height parent and
`.viewer-side-body` (`flex: 1 1 0`) computed to **0 px** — the panel content existed but `overflow: auto`
clipped it to nothing.

## What shipped (`src/pages/ViewerPage.tsx`, `src/app.css`)

Gave the drawer a deterministic flex chain so the body fills the panel height:

- Added a `content` className to the mobile `Drawer` (`classNames={{ content: 'aside-drawer-content', body:
  'aside-drawer-body' }}`).
- CSS: `.aside-drawer-content { display: flex; flex-direction: column }` (the Mantine content isn't a column
  by default); `.aside-drawer-body { flex: 1 1 auto !important; min-height: 0; display: flex; flex-direction:
  column }`; `.aside-drawer-body .viewer-side { flex: 1 1 auto; min-height: 0 }`.

After the fix the drawer body is full height, `.viewer-side-body` is ~568 px, and the peek/preview (and the
Notes/Search/Songs panels) render. Verified via Playwright on an iPhone 13 viewport: clicking a commentary
opens the drawer with the Preview tab showing the linked text (or the "Download to preview" prompt for an
undownloaded commentary). Desktop is untouched (the `.aside-drawer-*` rules only apply inside the drawer).

## Not changed (investigated, deemed not-broken / inherent)

While chasing this I measured the other two reported complaints on the deployed app:

- **"Catalog slow on each reload"** — a warm reload is ~315 ms and re-fetches only a 79-byte `manifest.json`
  (the catalog stays in OPFS; it is **not** re-downloaded). Not a network problem; any remaining slowness is
  mobile-CPU worker-boot + tree render. Left as-is.
- **"Downloads slow on mobile"** — slices *are* well-compressed (Genesis = 4.6 MB gz → 22 MB, ~5×; it bundles
  5 editions). The cost is inherent data size + decompress/merge of ~22 MB on mobile CPU, not a compression
  bug. Left as-is.
