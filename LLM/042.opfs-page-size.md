# 042 — Slow book downloads on incognito / Firefox: 64 KB pages

**Date:** 2026-06-24

## Why

Downloading a book felt **snappy in the installed PWA / a normal tab, but very slow in an incognito tab and
on Firefox**. The user guessed it was OPFS-related — correct.

Measured (same bytes, no network throttle, so only the OPFS work is timed):

| context | book merge phase |
|---|---|
| persistent storage (normal tab / PWA) | ~620 ms |
| non-persistent (incognito) / Firefox   | ~2,900 ms (~4.5×) |

The **download** isn't the difference (same bytes); the **merge** is. And it isn't fsyncs
(`PRAGMA synchronous=OFF` changed nothing) nor cache spill (a 256 MB `cache_size` helped only ~15%).

## Root cause

A book's cost is its **cross-reference links**, not its text: a Genesis-class slice is ~4.6k content rows but
**~155k link rows**, and the client `links` table carries `UNIQUE(from,to)` + a from-index + a to-index.
Merging therefore does **~465k B-tree inserts scattered across the file**. At the default **4 KB page size**
that's ~9,000+ dirty pages → thousands of small, random OPFS page-writes at COMMIT. The initial streamed
`importDb` is fast (few large sequential writes); the merge is slow because non-persistent / Firefox OPFS is
slow **per write op**, and there are thousands of them.

## What shipped (`src/db/worker.ts`)

Re-page the content DB to the **64 KB** max page size, which means ~16× fewer (larger) page-writes per merge.
`repageTo64k()` runs once on a **fresh** boot — `PRAGMA page_size=65536; VACUUM;` — while the DB is still just
the small catalog (no books merged yet), so it's **negligible** at boot (measured 2,689 ms vs 2,691 ms
without it). Wired into both fresh-boot paths (normal + corrupt-cache self-heal); guarded to a no-op when the
DB is already 64 KB; never re-pages an existing (possibly multi-GB) persistent cache.

**Result (incognito, 3 books):** each merge ~1,100–1,260 ms, down from ~2,900 ms — **~2.5× faster** — with
data intact (verified content + links rows, `page_size=65536`). No data loss, no feature loss, client-only
(works on the *current* deployed corpus — no republish), and it helps Firefox too.

## Not done (deliberately)

- The **download** size is unchanged — still links-dominated (~2.5 MB of the 4.6 MB gz). Cutting that is a
  separate, bigger call (defer/lazy-load links, or prune the auto "reference" links) — left for the user to
  decide.
- The slicer could ship the boot DB at 64 KB so even the one-time client VACUUM becomes unnecessary; deferred
  (needs a corpus republish; the client `repageTo64k()` already covers fresh downloads on today's data).
