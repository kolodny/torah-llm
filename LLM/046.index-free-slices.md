# 046 — Index-free slices (smaller downloads)

**Date:** 2026-06-26

## Why

Per-book slices were the dominant mobile download cost, and ~half of a slice's bytes were a **UNIQUE
auto-index it never needs**. A slice is write-once transport: the client merges its rows into the main OPFS DB
(which keeps its own constraints + indexes) and never queries the slice itself. Yet each slice shipped the
auto-indexes created by the table-level `UNIQUE(...)` constraints — for `links`, `UNIQUE(from_id, from_ref,
to_id, to_ref)` alone is ~9.6 MB of a 22 MB Genesis slice.

The existing "index-free" step in `slice-db.ts` only stripped explicit `CREATE INDEX` lines; it missed the
inline `UNIQUE` constraints (which auto-create indexes). (This is the same idea I'd floated and reverted
earlier — now done deliberately, with measurement.)

## What shipped (`ingest/slice-db.ts`)

`SLICE_SCHEMA` now also strips the inline `UNIQUE(...)` table constraints, removing the comma + the clause
together so each `CREATE TABLE` stays valid:

```
const SLICE_SCHEMA = SCHEMA_SQL
  .split('\n').filter((l) => !/CREATE (UNIQUE )?INDEX/i.test(l)).join('\n')
  .replace(/,\s*UNIQUE\s*\([^)]*\)/gi, '');
```

`content.id` / `links.id` stay `INTEGER PRIMARY KEY` (rowid alias, no extra index); the tiny `TEXT PRIMARY KEY`
auto-indexes on `editions`/`meta`/`toc` are left as-is. The boot DB still uses the full `SCHEMA_SQL` (it keeps
its toc index). The master has no duplicate rows, so a slice needs no `UNIQUE` to dedup, and the client's
`INSERT [OR IGNORE]` merge relies on the **main** DB's constraints, not the slice's.

## Measured (Genesis, full master: 5 editions, 158k links)

- rebuilt **with** the UNIQUE indexes → 4.72 MB gz (`sqlite_autoindex_content_1` + `sqlite_autoindex_links_1`
  present)
- rebuilt **index-free** → **2.59 MB gz** — those two auto-indexes gone → **~45% smaller**.

End-to-end: a fresh dev client downloaded + merged the index-free Genesis slice — **6,759 content rows and the
Gen 1:1 links (1,646) intact and queryable, and the main DB still has its 3 link indexes** (rebuilt locally).
`content_version` is row-based, so it's unchanged — existing clients keep their current slices; **new**
downloads get the smaller ones. The savings land on the next full corpus re-slice / republish (an app-only
deploy reuses the cached corpus and won't re-slice).
