# 055 ‚Äî External plugin bundles + a nicer plugin API surface

**Date:** 2026-06-29

> Status: **uncommitted, under review** (the user asked to hold the plugin-build/API work for a single
> source-of-truth review before committing). This entry documents it so the thread isn't lost.

## Why

Plugins lived as in-tree modules auto-loaded by a Vite glob ‚Äî fine for first-party, but not how a *third
party* would ship one. We wanted **one way** to write a plugin (first-party and external identical), with a
self-contained, copy-pasteable contract. And once the loader was generic, the per-plugin `ctx` had grown
warts: `contribute('viewer', '<stringy-slot>', ‚Ä¶)` everywhere, the cross-plugin Code API leaked onto the
global runtime, and every result link hand-rolled the same `href + modifier-aware onClick` boilerplate.

## What shipped

### External plugin bundles (IIFE + `window.__torahRuntime`)
- A plugin is now **one IIFE `.js`** built by `vite.plugin.config.ts` (repo root, next to `vite.config.ts` ‚Äî
  it's a Vite config; see the placement note below). `react` / `react/jsx-runtime` / `@mantine/core` /
  `react-router` are **externalized** to `window.__torahRuntime.*` so a plugin shares the host's *single*
  instance (hooks/JSX work) and stays ~1‚Äì10 KB.
- **`src/plugins/Plugin.type.ts`** ‚Äî the whole contract, fully self-contained (only depends on `react`'s
  types). Copy it into an external project for full typing. It also declares `window.__torahRuntime`.
- **`src/plugins/runtime.ts`** ‚Äî `installPluginRuntime()` sets `window.__torahRuntime` once at startup
  (called from `src/main.tsx`), before any plugin loads.
- **`host.tsx`** ‚Äî `discover()` is shared by both load paths; `registerExternalPlugin()` +
  `loadExternalPlugins(base)` (serial `<script>` injection, reads `window.__torahPlugin.default`).
  `loadPlugins()`'s glob is narrowed to **only** `code-search` (it pulls in Monaco ‚Üí stays bundled); every
  other plugin builds to `public/plugins/<id>.js` and loads the same way a third-party plugin would.
- **`scripts/build-plugins.mjs`** (`npm run build:plugins`) builds `plugins/*` (except code-search) ‚Üí
  `public/plugins/<id>.js` + `index.json`. `public/plugins` is **gitignored** (built artifact); the
  `publicDir:false` in the plugin config stops Vite copying the 886 MB `public/db` corpus into the bundle dir.
- **`PLUGINS.md`** is the single source-of-truth doc; **`examples/`** removed.

### Nicer `ctx` surface (the 4 cleanups ‚Äî "do them all")
1. **`ctx.viewer.*`** ‚Äî typed sugar over `contribute('viewer', ‚Ä¶)`: `addSidebar`, `addVerseAction`,
   `addLinkAction`, `addDecoration`, `addEditor`, `addTextSelectAction`. All 7 plugins migrated off the
   stringy `contribute('viewer', '<slot>', ‚Ä¶)`.
2. **`ctx.exposeApi(name, api)` / `ctx.getApi(name)`** ‚Äî a flat cross-plugin registry (in `host.tsx`, entry
   removed when the publisher unloads). **code-search** now `exposeApi('code-search', codePageApi)` (the
   *factory*, so each consumer's renderers/fns/samples track under *its own* ctx); **gematria** + **torah-code**
   acquire it with `ctx.getApi<CodePageApiFactory>(CODE_PAGE_ID)?.(ctx)` and only **type**-import from
   `code-search/api`. Replaces the old `codePageApi`-on-the-runtime leak.
3. **`window.__torahRuntime` reorganized** ‚Äî top level is just the shared libs
   (`react`/`jsxRuntime`/`mantine`/`reactRouter`, what the build externalizes); the host SDK moved under
   **`.sdk`**: `definePlugin`, `registerPlugin`, `components.{BookCheckTree}`, `util.{stripHtml}`,
   `hooks.{useSlot,usePages,useDecorationsTick}`. `codePageApi` dropped from the global.
4. **`ctx.ui.linkProps(book, ref)`** ‚Üí `{ href, onClick }` to spread on an `<a>`/`<Anchor>`: left-click
   SPA-navigates, cmd/ctrl/shift/middle falls through to the browser (real new-tab). DRYs the 5 duplicated
   link sites (gematria, pasuk-name, notes, search, torah-code). `ctx.ui.href` kept.

### `vite.plugin.config.ts` placement
Stays at **repo root** next to `vite.config.ts` (both are Vite configs; `build:plugins` invokes it from root;
a build config doesn't belong in the app `src/` graph ‚Äî it imports `vite`). The copy-paste "kit" cohesion is
served by `PLUGINS.md` naming both files an author needs (`Plugin.type.ts` + `vite.plugin.config.ts`).

### Production build + offline (fixed ‚Äî externalizing the plugins broke both)
Moving plugins out of the app bundle into runtime-loaded `public/plugins/*.js` exposed a packaging gap:
`vite.config.ts` sets `copyPublicDir:false` (so the 886 MB `public/db` isn't copied on every app build), so
`public/plugins/*` **never reached `dist`** ‚Äî the dev server only worked because it serves `public/` directly.
A production build (and therefore the deployed site) would have had **no external plugins at all**, online or
off. Fixes:
- `package.json` `build` now runs `build:plugins` and copies `public/plugins ‚Üí dist/plugins` **before**
  `vite build`, so vite-plugin-pwa globs them into the precache manifest.
- `vite.config.ts` `globPatterns` adds `plugins/index.json` ‚Äî the `.js` bundles were already covered by
  `**/*.js`, but the loader manifest is JSON and was being skipped, so offline plugin discovery would fail.
- **Also fixed a pre-existing boot-DB precache miss** found while checking the deploy: the live
  `sw.js` precache (kolodny.github.io/torah-app) had the app shell but **not** `db/db.sqlite.zst` /
  `db/manifest.json`, because the deploy runs `npm run build` (workbox glob) *before* `npm run build:db`
  (which rsyncs the corpus into `dist/db`) ‚Äî so at glob time the boot DB wasn't there yet (the
  `globPattern db/db.sqlite.zst doesn't match` warning was the tell). The `build` script now copies the
  ~1.2 MB boot DB + manifest into `dist/db` **before** `vite build`, realizing 033's intended
  "catalog resolves offline on a fresh/evicted OPFS". The deploy is unchanged ‚Äî it already uses
  `npm run build` + `npm run build:db`, which is exactly where the fix lives.

Verified offline for real (Playwright): built `dist`, served via `vite preview`, let the SW install +
precache (38 entries incl. all 7 plugin bundles + `plugins/index.json` + the boot `db.sqlite.zst`), then
**killed the preview server** and reloaded ‚Äî the app booted from cache, the SDK initialized, and the external
**Gematria** + **Name Pasuk** pages (plus bundled Code) registered, 0 console errors.

### Activation race on cold deep-link (fixed)
External bundles load **async** (serial `<script>` injection), so a plugin could be discovered *after* its
activation event already fired ‚Äî e.g. a cold deep-link to `?book=Genesis` sets the reader (firing `onBook:*`)
before `notes`/`search`/`songs`/`reader-modes` arrive, so they never activated (the `lastActivatedBook` guard
means `onBook:*` won't re-fire). Fix in `host.tsx`: a module-level **`firedEvents` Set** records every event
`fireActivation()` fires; `registerExternalPlugin` now activates a just-discovered plugin if it's startup-ready
**or any event it declared has already fired** (`shouldActivateOnDiscovery`). Verified: the cold deep-link now
shows Notes/Search/Songs sidebars + both verse actions (Gematria + Add note), 0 errors.
