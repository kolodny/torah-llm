# 012 — Multi-source ingestion

**Date:** 2026-06-09

Generalized the Sefaria-only pipeline so multiple corpora can be imported, and added a second
source (Open Scriptures / WLC) to prove it.

## Source survey

- **HebrewBooks.org** — a PDF archive. Its GitHub org has only metadata/app repos, no clean
  structured full-text corpus → not suitable for text import.
- **Open Scriptures Hebrew Bible** (`openscriptures/morphhb`) — WLC in OSIS XML per book; WLC text
  is **Public Domain** (morphology CC-BY 4.0). **Chosen** as source #2.
- **otzaria** (`Sivan22/otzaria`, per the owner's pointer) — a reader app that credits Sefaria,
  **Dicta**, **Orayta**, etc.; stores texts as TXT/Docx/PDF in a *separate* library repo
  (`zevisvei/otzaria-library`). It does **not** ingest HebrewBooks as structured text. Good
  candidate future sources (TXT/DOCX adapters): `Dicta-Israel-Center-for-Text-Analysis/Dicta-Library-Download`,
  `zevisvei/otzaria-library`, `MosheWagner/Orayta-Books`.

## Architecture (rewired Sefaria-only assumptions)

- **Source dimension:** `toc.id` is now namespaced `"<source>:<natural>"` (e.g. `sefaria:Genesis`,
  `oshb:Genesis`) beneath a per-source **root** node; added a `toc.source` column. Ids stay stable
  string keys, now collision-free across corpora. `BOOT_VERSION` → 3 (self-heals old boot DBs).
- **Adapter interface** (`scripts/sources/types.ts`): `SourceAdapter { id, name, fetchSubset(),
  ingest(ctx) }`; `IngestCtx` emits `toc/content/meta/link` rows. Adapters insert directly through
  prepared statements (scalable), `ingest` is synchronous (reads local files).
- **Adapters** (`scripts/sources/`): `sefaria.ts` (fetch + ingest + the Sefaria-specific ref-path
  flattening, moved here; emits namespaced ids + source root + derived commentary links),
  `oshb.ts` (downloads OSIS XML; a small regex parser reconstructs verse text — strips the `/`
  morphology separators, joins maqqef with no space, appends sof-pasuq), `index.ts` (adapter list).
- **build-master.ts** now just owns the DB + prepared inserts and runs each adapter; the
  `has_content`/`content_rows` pass is source-agnostic SQL.
- **Slice filenames** switched to `toc_${encodeURIComponent(id)}.sqlite` (+ a second encode for
  the URL) so ids containing `:` / `/` are filesystem-safe (macOS mangles `:`) and injective.
- **Browser:** generic — the catalog tree groups by the source-root nodes automatically; added
  `source` to `TocRow`.

## Verified (Playwright MCP)

- Catalog shows two source roots: **Sefaria** (Tanakh→Torah + Rashi) and **Open Scriptures (WLC)**
  (Genesis/Jonah/Ruth). 13 content-bearing books.
- **No collision:** `sefaria:Genesis` (he+en, with Sefaria `<big>` markup) and `oshb:Genesis`
  (WLC, Hebrew-only, no markup) both load independently.
- **On-demand across sources:** `oshb:Jonah` fetched `toc_oshb%253AJonah.sqlite [200]` (the
  double-encoded namespaced filename resolved) and showed the exact WLC text
  (`וַֽיְהִי֙ דְּבַר־יְהוָ֔ה אֶל־יוֹנָ֥ה בֶן־אֲמִתַּ֖י לֵאמֹֽר׃`). 0 errors. Build passes.

## Source #3 — Orayta (TXT)

Added `scripts/sources/orayta.ts`: fetches book `.txt` files from `MosheWagner/Orayta-Books`
(free-licensed, "Mainly GPDL and Creative Commons"). Orayta's format uses line-leading markers
`$` (title) / `^` (section/parsha) / `~ … פרק-<n>` (chapter) / `! {<gematria>}` (verse); the
adapter numbers chapters/verses sequentially (→ refs `c:v`) and strips Orayta's inline aliyah
HTML (`<BR><span class="Aliyah">…</span>`). Subset: **Genesis** (a third coexisting edition),
**Song of Songs**, **Esther**. Verified in-browser: Esther loads on demand (167 verses · 10
chapters, clean text), 0 errors. Now **3 sources, three "Genesis" editions** coexist.

## Robustness note (stale browser state)

The worker architecture changed several times this session; a browser holding stale OPFS/
SharedWorker/Web-Lock state could get stuck on load. Mitigations: the boot DB self-heals on a
`BOOT_VERSION` bump, and the client now **times out port acquisition (15s)** with an actionable
error instead of hanging silently. Recovery if stuck: close all tabs of the app and reload, or
clear site storage (DevTools → Application → Clear storage), or use "Wipe local DB".
