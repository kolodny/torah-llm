# 056 — Gematria marker bug, crown-letters sample, RTL rendering audit

**Date:** 2026-06-30

## Why

Three threads from using the live app:
1. A bug report: the Gematria **whole-verse search** for value 1000 over Tanakh missed **Psalms 59:18**
   (עֻזִּי אֵלֶיךָ אֲזַמֵּרָה… — gematria exactly 1000).
2. The Code-page "shatnez" sample was framed wrong (it found a word *containing* the letters שעטנז); the real
   idea is **crown letters (תגין)** — the seven שעטנז״גץ that get the triple-crown in a sefer Torah.
3. The Gematria result rows (and a couple of other surfaces) rendered Hebrew with broken RTL/truncation.

## What shipped

### Gematria marker bug (Psalms 59:18)
Psalms 59:18 is stored `…חַסְדִּֽי׃&nbsp;<span class="mam-spi-pe">{פ}</span><br>`. The gematria **search page**
used its own DOM-`textContent` strip, which dropped the `<span>` tag but **kept the literal `{פ}`** → counted
פ=80 → 1080, not 1000, so the verse was missed. (The SQL `gematria()` function was already correct from
[052](052.cleanup-functions-corpus-audit.md); only the plugin page kept a naïve strip.) Fix: the page now uses
the host's shared `stripHtml` (`sdk.util.stripHtml`), which drops `mam-spi`/footnote markup exactly like the
corpus. Verified in-browser: the search now returns 5 matches incl. Psalms 59:18.

### Crown-letters (תגין) Code-page sample
Rewrote the sample to count crown-letters per Tanakh word (set ש ע ט נ ז ג צ incl. finals ן/ץ) and rank them.
Verified against the full canonical Tanakh **in the actual browser** (downloaded all 39 books to OPFS, ran via
the Code page): the record is **five**, held by exactly two words — **שעטנז** (the forbidden mixture, Lev 19:19 &
Deut 22:11) and **שַׁעַשְׁגַז** (Shaashgaz, a chamberlain in Esther 2:14). The sample scopes to the canonical
Tanakh via `toc.parent_id` (independent of the SELECTED chips), since the payoff lives in Ketuvim.

### RTL / truncation rendering audit
Swept every Hebrew-text render site. Most were already direction-aware (viewer segments + flow + PeekPanel set
`dir` per edition lang; SQL Code page cells/links use `dir="auto"`; reader-modes/torah-code/pasuk-name set
`dir`). Fixed the three that weren't:
- **Gematria result row** — was a left-aligned LTR line that clipped verses mid-word. Now a flex row: the ref
  then the verse sitting **right next to each other** (`flex: 0 1 auto`, no whitespace gap), RTL, single-line
  with a CSS ellipsis when too long (full verse kept; CSS clips at the correct end). `.gem-hit*` in `app.css`.
- **Search snippet** — had no `dir` → Hebrew rendered LTR-jumbled. Added `dir="auto"` (verified resolves to
  `rtl` for Hebrew snippets, `ltr` for English books).
- **Notes body** — added `dir="auto"` (notes can be either script).

All verified in the Playwright browser against the real downloaded corpus.
