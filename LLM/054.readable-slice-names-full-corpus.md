# 054 — Readable slice filenames + ship the whole corpus (886 MB on Pages)

**Date:** 2026-06-29

## Why

Two follow-ups to the [zstd switch](053.zstd-slices.md): (1) slice filenames were percent-encoded toc ids,
double-encoded again in the fetch URL (`%2520` for a space) — a cross-host footgun and the source of 23k
special-char names; (2) now that zstd fits the full corpus under Pages' 1 GB, stop shipping only a subset.

## What shipped

- **Readable slice filenames** (`shared/slice-path.ts`). `sliceFileName(id)` is now the id slugged to
  `[A-Za-z0-9_]` (non-alphanumeric runs → `_`, trimmed, capped 180) + an 8-hex of the 53-bit id hash:
  `A_Jewish_Critique_of_the_Philosophy_of_Martin_Buber_Postscript_14411137.sqlite`, `Genesis_18b8f554.sqlite`.
  ASCII-only → `sliceUrlPath` no longer percent-encodes (kills the double-encoding). The hash makes it
  **injective**: a bare slug collides on **2 real book pairs** that differ only by a trailing `*`
  (`Sifra, Tazria Parashat Negaim, Chapter 2` vs `…Chapter 2*` — confirmed in Sefaria's own source data, two
  distinct sections, 20 vs 34 rows), which would otherwise silently drop a book. The hash only has to separate
  same-slug ids, so 8 hex is ample. The slicer now **asserts** filename uniqueness at build time (tripwire).
- **Full corpus** (`ingest/build-publish-master.ts`): emptied `EXCLUDE_TOP_CATEGORIES` (was Halakhah +
  Second Temple, dropped only to fit gzip) — the whole corpus now ships.

## Verified

- `sliceFileName` injective over all **23,444** ids (0 collisions); the 2 `*` pairs split cleanly
  (`…Chapter_2_0cbe60fe` vs `…Chapter_2_0f5c3e22`).
- Full-corpus re-slice → **public/db = 886 MB** (841 MB slices + 1.14 MB boot `db.sqlite.zst` + manifest),
  **23,444 books, under 1 GB with ~138 MB headroom.**
- In-browser (wipe → fresh boot): catalog loads 23,444 books from `db.sqlite.zst`; `ensureBook('Genesis')`
  fetches `Genesis_18b8f554.sqlite.zst`, fzstd-decodes, merges → Gen 1:1 = 28 letters, content correct.
- `npx tsc` clean. (`public/db` gitignored; the publish pipeline re-slices on deploy.)
