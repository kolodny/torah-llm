# 026 — Plugin platform

**Date:** 2026-06-10

## Vision
Make this a **Torah platform**: the core is just the **database + base renderer**, and everything
else is a plugin. Plugins must integrate at *every* level — from augmenting a single link, to owning
the rendering of a whole view — and interact with core data and UI seamlessly.

## Model
- A plugin is a module at **`plugins/<id>/index.tsx`** that default-exports
  `definePlugin({ id, name, activate(api) })`. **Auto-discovered** via Vite `import.meta.glob` — drop
  a folder in `plugins/`, no registry edits.
- `activate(api)` reads/writes data and **registers contributions**. The host collects them into a
  registry the app's UI renders.

## The API (`src/plugins/types.ts`, host `src/plugins/host.tsx`)
The `api` handed to `activate` is the host + `add*` methods:
- **`data`** — `getToc/getEditions/getContent/getSegment/getSiblings/getLinks/ensureBook` + raw
  read `query(sql, params)`. The same queries the core uses.
- **`storage`** — persistent, **plugin-scoped** KV (`get/set/delete/keys`), namespaced in
  `localStorage` → survives the content-DB wipe lifecycle. (Notes live here.)
- **`ui`** — `navigate / peek / openPanel / closePanel / toast`.
- **reader context** — `reader()` snapshot + `onReaderChange(fn)` (book, ref, available + selected
  editions). UI actions are **late-bound**: a plugin activates once at startup; the React
  `PluginProvider` installs the real `navigate/peek/openPanel/toast` on mount.

## Contribution hooks (the integration levels)
| Hook | Level | Example |
|---|---|---|
| `addCommand` | global action | "Search in this book" |
| `addPanel` | right-rail UI | Search / Notes / Songs panels |
| `addVerseAction` | augment a verse | "Gematria", "Add note" (the verse ⋯ menu) |
| `addLinkAction` | augment a link | "♪ Songs" on a cross-reference |
| `addTextDecorator` | transform verse HTML | highlight gematria, tap-a-word *(hook defined; no v1 plugin uses it yet)* |
| `addView` | **own the rendering** | a custom reader/mode when `when(reader)` matches *(hook wired; no v1 plugin yet)* |

The app reads the registry to render: a commands menu + panel rail (header), a panel host (the right
column — peek and plugin panels share it), a `⋯` actions menu per verse, link-action buttons on each
cross-reference, an optional view override, and toasts. (`src/plugins/ui.tsx`.)

## v1 example plugins (in `plugins/`)
- **gematria** — a verse action (a verse's total value; Genesis 1:1 = 2,701 ✓) **and** a panel that
  searches the current book by gematria value (words or whole verses → jump to a hit; verified: 205
  words = 86 incl. אלהים/מבדיל). *Your "search gematria" usage.*
- **search** — panel + command; phrase search across the current book → jump to a hit (verified: 53
  hits for אֱלֹהִים, click → navigate). *Your "search a phrase" usage.*
- **notes** — verse action ("Add note") + panel, on persistent plugin storage. *Your "note taker."*
- **songs** — link action ("♪ Songs") + panel; opens for the linked passage. *Your "song explorer"
  usage* (the link→plugin flow is wired; a real songs dataset is the data half).

## Verified (Playwright)
4 plugins auto-loaded; commands menu + 3-panel rail + a ⋯ menu on all 1,533 Genesis verses; gematria
toast (2,701); search query + navigate; songs link action opened its panel for the target ref. Prod
build green (bundle 100 KB gzip).

## Next / known gaps
- **Decorators** and **view-override** hooks are defined + wired but no v1 plugin uses them — they're
  the path for tap-a-word/lexicon and "own the rendering" (e.g. a continuous-scroll or interlinear
  reader). 
- **Storage** is localStorage (fine for notes); a large/structured plugin store could move to a
  dedicated never-wiped OPFS DB.
- **Search / gematria search** run over the *current* book (in JS / `LIKE`). Multi-book "across
  selected books" + a prebuilt index is the follow-up (and would extend to cantillation search).
- **Songs** needs a real dataset keyed by ref/phrase. A plugin could ship its own data table.
- Panels currently share the single right column with the peek; a multi-panel layout is possible later.
