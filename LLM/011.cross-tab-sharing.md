# 011 — Cross-tab data sharing (leader-tab + broker)

**Date:** 2026-06-09

Implements true cross-tab sharing (chosen approach from [010](010.followups.md): leader-tab +
SharedWorker port broker). No cross-origin-isolation headers; deploys anywhere.

## Why this works when the SharedWorker-pool plugin couldn't

Two browser facts blocked the obvious designs: a SharedWorker can't use OPFS sync access handles,
and **Chrome won't let a SharedWorker spawn a `Worker`** (`Worker is not defined`). This design
sidesteps both:

- The dedicated db worker (which owns the SAH pool) is spawned by the **leader TAB** — a window
  context, where `Worker` exists.
- The **SharedWorker only brokers MessagePorts** between tabs; it never spawns a worker, so it
  never touches the missing API.

## Architecture

- **Election:** every tab calls `navigator.locks.request('torah-leader', {exclusive})` with a
  never-resolving callback. The holder is the leader; closing its tab releases the lock and
  promotes another tab.
- **Leader** (`client.ts`): spawns the dedicated `worker.ts`, and on each broker `mint` request
  creates a `MessageChannel`, hands one end to its worker (`{type:'connect'}` → `expose`) and
  returns the other to the broker.
- **Broker** (`broker.ts`, SharedWorker): every tab sends `want`; the broker asks the leader to
  `mint` and forwards the returned port to the tab. On a new leader (handoff) it broadcasts
  `leader-changed`.
- **Every tab** (leader included) talks to the one worker over its minted port via comlink, so
  all tabs share one connection + one OPFS pool. `withApi()` retries once on `leader-changed`.
- **Self-booting worker** (`worker.ts`): downloads the TOC DB if absent, opens it, re-downloads
  on a `user_version` bump — so any tab's first query just works regardless of who's leader.

## Files

- `src/db/worker.ts` — dedicated db engine (SAH pool); self-boots; exposes API per connected port.
- `src/db/broker.ts` — SharedWorker that brokers ports (no DB, no worker spawning).
- `src/db/client.ts` — leader election + broker handshake + reconnect; same public query helpers.

## Verified (Playwright MCP)

- **Share A→B:** tab 0 downloads Genesis; tab 1 opens Genesis and shows "✓ local" with **0**
  `db/*.sqlite` requests (reads tab 0's data).
- **Share B→A + routing:** tab 1 opens Exodus — the fetch happens in the **leader's** worker
  (tab 0's network shows `toc_Exodus.sqlite`, tab 1's shows none); tab 0 then reads Exodus with 0
  fetches.
- **Handoff:** close the leader (tab 0); the survivor becomes leader and loads a new book, with
  Genesis + Exodus + Leviticus all still "✓ local" (shared OPFS pool persisted). 0 errors/warnings.

## Known limitation

A single in-flight call at the exact moment the leader tab closes may need one retry (the next
user action reconnects automatically via `leader-changed`). New work after handoff is unaffected.
