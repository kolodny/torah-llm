# 001 — Investigation of the abandoned reference

**Date:** 2026-06-09
**Reference (READ-ONLY, never modified):** `/Users/mkolodny/projects/torah`

## What the reference contained

A React + Vite + TypeScript app plus a Node ingestion pipeline:

- `scripts/build-db.ts` — read raw Sefaria-Export flat files → one `db/master.sqlite`.
- `scripts/slice-db.ts` — split master into `public/db/toc_<id>.sqlite` chunks + a small
  `public/db/db.sqlite` holding just the table-of-contents.
- `scripts/organize-sefaria.ts`, `scripts/utils.ts` — flatten nested text JSON into
  `ref → text` rows using each text's schema (`sectionNames`, `index_offsets_by_depth`,
  complex `nodes`). The `createRefPath`/`format` logic here is the non-trivial, valuable part.
- Browser: `src/orm/worker.ts` (SQLite-WASM in a Web Worker, OPFS SAH-pool VFS, exec/upload/
  merge via comlink) and `src/orm/index.ts` (drizzle `sqlite-proxy` over the worker).
- Drizzle ORM + drizzle-kit migrations; MUI; react-query; react-router; use-query-params.

## The breakage (why it was abandoned)

Uncommitted edits switched `orm/schema.ts` TOC ids from `integer` → `text`, but
`build-db.ts` still assigned ids via `lastInsertRowid` (an integer rowid). With a TEXT
primary key that leaves `id` NULL → the hierarchy (`parent_id` lookups) and links break.

Confirmed by inspecting `db/master.sqlite` (built earlier with integer ids) and the
drizzle migration (`integer PRIMARY KEY`) vs. the working-tree `schema.ts` (`text`).

## Reusable knowledge taken into the rebuild

- The OPFS SAH-pool worker pattern (no COOP/COEP headers needed).
- The `merge` approach: `ATTACH` a downloaded slice and `INSERT OR IGNORE` its rows.
- The ref-path flattening algorithm (rewritten cleanly in the new `scripts/`).
- Slice-per-book chunking keyed by TOC id.
