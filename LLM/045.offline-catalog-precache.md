# 045 — Offline: precache the boot catalog so it always resolves

**Date:** 2026-06-25

## Why

Offline, the app shell loaded (precached by the PWA) but **the catalog never resolved**. The common case
actually works — if the boot DB is already in OPFS, `bootSequence` opens it and `reconcile`'s `manifest.json`
fetch failing offline is caught (`catch { return }`), so the cached catalog is used (verified on deployed:
offline reload resolves in ~300 ms). The failure is when the **boot DB isn't in OPFS** (fresh install, evicted
cache, or the user went offline before the first boot-DB download finished): `bootSequence`'s `fresh` path
calls `streamInto(BOOT_PATH, db.sqlite.gz)`, which fails offline and rethrows (it's a network error, not a
corrupt image) → the `getToc()` query rejects → catalog never loads.

Root cause: the PWA **excluded all of `db/`** from precache (`globIgnores: ['**/db/**']`) — including the tiny,
essential boot catalog. Only the per-book slices (multi-GB) should be on-demand.

## What shipped (`vite.config.ts`)

Precache the **boot catalog** alongside the app shell, keep the per-book slices on demand:

```
globPatterns: [... , 'db/db.sqlite.gz', 'db/manifest.json'],   // ~1.5 MB boot DB + 79 B manifest
globIgnores: ['**/db/toc_*'],                                   // the 23k per-book slices stay on-demand
```

`maximumFileSizeToCacheInBytes` already allows 4 MB (boot DB is ~1.5 MB).

## Verified

- Build with a stub `db/` (real boot DB + fake slices): the generated `dist/sw.js` precache manifest **includes
  `db.sqlite.gz` + `manifest.json` and excludes every `toc_*`**.
- `vite preview` + Service Worker, then `setOffline(true)`: `fetch('db/db.sqlite.gz')` → **200 (from the SW
  precache)**, `manifest.json` → 200, a per-book slice → `Failed to fetch` (correctly not precached). So the
  offline `streamInto(db.sqlite.gz)` boot now succeeds and the catalog resolves; books still download
  on-demand (and fail gracefully offline if not yet cached).

Takes effect on the next deploy (the SW is rebuilt with the new precache manifest). The boot DB + manifest are
a matched snapshot, precached together, and updated together when a new publish rebuilds the SW (autoUpdate).
