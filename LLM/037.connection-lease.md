# 037 — Connection lease: per-page worker, release on inactive (kills the reload lockup)

**Date:** 2026-06-23

## Why

Even after the SAH-handle fixes ([033](033.offline-and-reliability.md)), reloading an **inactive**
(backgrounded) tab still locked up badly. Root cause was structural: the OPFS SAH pool's exclusive
SyncAccessHandles were released only when the owning context was *destroyed*, and the browser delays
destroying a frozen/inactive page (Page Lifecycle). The old design made this worse — a **SharedWorker
broker + a Web-Locks leader election** spawned a single shared db worker; on reload the new worker
queued behind the old (frozen) one's handles. Lots of coordination, and the handles still lingered.

The request: stop coordinating workers. Have the active page hold the connection directly and only
release it when it stops being active. Plus: *is connect/query/disconnect fast enough to just grab the
connection around each query?*

## Measurement first (sqlite-wasm 3.53, Chromium, uncontended)

The pool exposes `pauseVfs()` (sync — release the SAHs; db must be closed first) and `unpauseVfs()`
(async — re-acquire) precisely for multi-tab handoff. Benchmarked:

| op | time |
|---|---|
| query (open db) | 2.3 ms |
| `pauseVfs()` (release handles) | 1.4 ms |
| `unpauseVfs()` (re-acquire handles) | 2.0 ms |
| reopen db + pragmas | 0.24 ms |
| **full release → re-acquire** | **~3.6 ms** |

So yes — grabbing/dropping the connection is cheap (~3.6 ms), and the primitive is `pauseVfs`/`unpauseVfs`,
**not** reinstalling the VFS per query (that's the slow ~tens-of-ms part).

## What shipped — the "connection lease"

Deleted the SharedWorker broker (`src/db/broker.ts`), the `torah-leader` election, and the
`torah-sahpool` lifetime-held lock. New model:

- **One dedicated worker per page** (`client.ts` just `new Worker(...)` + comlink `wrap`; `expose(api)`
  in the worker — no broker, no port brokering, no leader).
- The worker installs the SAH pool lazily and holds it only during a **lease**: every API call runs
  inside `withLease()`, which acquires the exclusive **`torah-db` Web Lock**, `unpauseVfs()`-es the pool,
  opens the db, runs, then — when no operation is in flight — schedules a release after a short idle
  (`LEASE_IDLE_MS = 600`, so back-to-back queries don't thrash pause/unpause). A ref-count (`leaseDepth`)
  guarantees we never release mid-operation (e.g. a long book merge).
- **Release on inactive:** the page calls `api.release()` on `visibilitychange→hidden` / `pagehide` /
  `freeze`. `release()` sets a flag and drops the lease as soon as nothing's in flight — closing the db,
  `pauseVfs()` (frees the handles), and freeing the Web Lock. So an inactive/backgrounded tab holds
  **nothing**, and reloading it (or another tab) acquires instantly. The next query re-acquires lazily.

Tabs no longer share one connection; they take turns via the lock. Correctness across tabs holds: the
lock serializes access, and each lease reopens the db, so every reader sees committed data. First boot
(download catalog / migrate in place / reconcile against the manifest) runs once per session; re-acquires
after a pause just reopen the connection. Corruption self-heal now happens in-lease (`wipeAndRebootInLease`).

## Verification (Playwright)

- **Rapid-reload SAH stress** (`repro-deploy.mjs` Test A): **0** `NoModificationAllowedError`; app boots
  every time. Downloads flat. (pagehide→release frees the handles before the next worker starts.)
- **Backgrounded-tab handoff** (ad-hoc): tab 1 reads, goes hidden (→releases); tab 2 opens and reads
  while tab 1 is hidden; tab 1 refocuses and re-acquires cleanly — 0 errors.
- **Pagination** (`repro-pagination.mjs`) end-to-end through the lease path: unchanged (Genesis opens at
  31 verses, grows on scroll). Responsive e2e: 7 passed / 1 skipped.

## Trade-offs / notes
- A long single operation (e.g. a multi-second book download) holds the lease for its duration, so a
  second tab waits until it finishes. Acceptable (correctness > concurrency here); revisit by chunking if needed.
- True side-by-side visible tabs both want the lock; the FIFO lock serializes them (one waits up to one
  idle window). No corruption, just a brief wait — far better than the old lockup.
- `installPoolWithRetry` stays as a backstop, but conflicts are now rare since SAH ownership ⊆ lock ownership.
