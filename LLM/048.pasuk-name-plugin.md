# 048 — "Pasuk for your name" plugin (end of Shemoneh Esrei)

**Date:** 2026-06-28

## Why

The widespread minhag at the end of Shemoneh Esrei (before *Oseh Shalom*) is to recite a pasuk that **begins
with the first letter of one's name and ends with its last letter**. With the corpus + the read-only `query()`
+ the core `letters()` SQL function already in place, this is a natural, fully self-contained **page plugin** —
no reader/DOM integration, just two letters in → matching pesukim out — and a good demonstration that the page
API can host a useful tool with zero core changes.

## What shipped

- **New plugin `plugins/pasuk-name/index.tsx`** — auto-discovered by the host's
  `import.meta.glob('/plugins/*/index.{ts,tsx}')`, so no registration wiring. Registers a page
  (`registerPage`, icon ✡, "Name Pasuk"). Permission `data:read`.
- **Two Hebrew-letter dropdowns** (Mantine `Select`, the 22 base letters) for first/last letter — chosen over a
  free-text name field (per request) so input is unambiguous. The last-letter match expands to the **final
  form** too (`מ`→`ם`, etc.), since pesukim end in final-forms.
- **URL-backed state**: first/last live in `?first=…&last=…` (via the app's `useSearchParams`), so a search is
  shareable and survives reload; on mount, if both are present the search **auto-runs**.
- **Results grouped by book** with per-book counts ("93 matching pesukim across 21 books" + `Psalms · 14`
  headers, sticky on scroll). Without grouping, Tehillim sits ~halfway down a flat 93-row list (Ketuvim follows
  all of Neviim) and reads as "missing".
- **Query.** One Hebrew edition per verse (`MIN`), matched by `substr(letters(c.text),1,1)` (first consonant)
  and `substr(letters(c.text),-1) IN (base, final)` (last consonant). Restricted to the **24 canonical Tanakh
  books** via `c.toc_id IN (SELECT id FROM toc WHERE parent_id IN ('Tanakh / Torah','Tanakh / Prophets',
  'Tanakh / Writings'))` — the three pesukim divisions, excluding Targum/Rishonim/Acharonim/Modern commentary
  that also live under the `Tanakh` root. The `toc` join also supplies `order_index`, so results sort in
  **canonical Tanakh order** (Genesis→II Chronicles) then **numeric chapter:verse** — not the `toc_id`/`ref`
  strings, which put Deuteronomy first and `10:1` before `2:1` (and buried Psalms near the bottom past
  Isaiah). Results link to the verse via `ctx.ui.navigate`. `LIMIT 300`.

## Verified

Ran the exact query against `data/master.sqlite` with the real `strip()`/`letters()` re-registered
(better-sqlite3):
- **First cut used a ref heuristic** (`ref NOT LIKE '%:%:%'`) — results were swamped by **Abarbanel** and other
  2-level-ref Hebrew commentaries. Root-caused: `rootOf(toc) === 'Tanakh'` spans **3,613** books (all
  commentary-on-Tanakh nests under it); the clean signature is the three divisions above → exactly **39**
  book rows, all `kind='book'`, no sub-categories.
- With the toc filter, samples return real pesukim, each correctly bounded: אברהם (א…ם) 266, משה (מ…ה) 93,
  שרה (ש…ה) 71, דוד (ד…ד) 4 — e.g. דוד → Numbers 2:18 `דֶּגֶל…אֶפְרָֽיִם …` ends in ד.

Also verified **in-browser via Playwright** (dev server, all 39 Tanakh books fetched via `__torahDb.ensureBook`):
for `מ … ה` the page returns 93 pesukim, **Psalms present (14)** in its Ketuvim position, dropdowns restore +
auto-run from the URL, and after dropping a stray nested `ScrollArea` there is a **single scroll container**
(`.plugin-page`, 0 nested) that reaches the last row. Reproduced the original symptom too: the pre-fix
`ORDER BY c.toc_id, c.ref` put Deuteronomy first and Psalms at **row 80/93**, matching the reported "no Psalms".

`npx tsc` clean. (Searches whatever Tanakh is downloaded locally; empty-state points the user to the Storage tab.)
