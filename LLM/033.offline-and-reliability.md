# 033 — Offline PWA + reliability hardening (corruption, resize, SAH-handle race)

**Date:** 2026-06-22

## Why

After the GitHub Pages publish ([032](032.github-pages-publish.md)), real-world use on the deployed
app surfaced four reliability problems — all reported against **https://kolodny.github.io/torah-app/**:

1. The app should work **fully offline** (and ideally as an installable PWA, so the browser won't
   evict the downloaded corpus from OPFS when disk runs low).
2. **DB corruption on reload-mid-download**: starting 5 chumash downloads and reloading after only the
   first finished left the DB `SQLITE_CORRUPT: database disk image is malformed`.
3. **Resize jank**: resizing the window with a book open caused severe performance issues.
4. **`NoModificationAllowedError`** on fast consecutive reloads
   (`createSyncAccessHandle ... Access Handles cannot be created if there is another open Access
   Handle ...`), plus perceived "really slow" downloads on the deployed site.

## What shipped

### Offline PWA (committed `4abc73e`)
- `vite-plugin-pwa` (Workbox `generateSW`): precache the app shell incl. `.wasm`/`.ttf`, **exclude
  `db/`** (the corpus is fetched on demand into OPFS, not the SW cache).
- `src/main.tsx` calls `navigator.storage.persist()` (granted readily once installed) so OPFS isn't
  evicted under storage pressure.
- Build script (`package.json`) copies `public/favicon.* public/pwa-*.png ...` into `dist/` before
  `vite build` (Vite's `copyPublicDir:false`) so icons are served **and** precached.
- Verified: killed the dev server, app still booted from SW cache + OPFS.

### DB corruption self-heal + crash safety (committed `2426b5a`)
- Root cause: a reload **mid-`mergeSlice`** (the one big `BEGIN…COMMIT` that swaps a book's editions/
  content/links) left the rollback journal in a state the next boot read as a malformed image.
- `src/db/worker.ts` `openAndReconcile` now sets `PRAGMA journal_mode = TRUNCATE` +
  `PRAGMA synchronous = NORMAL` — crash-safe against an app reload mid-write, measured cost only
  ~60–100 ms (NOT the cause of perceived slowness — see below).
- **Self-heal**: `isCorrupt()` detector + `wipeAndReboot()` (close → `pool.wipeFiles()` → reboot).
  `openAndReconcile` recovers on a corrupt boot; `exec` retries once through wipe+reboot on corruption.
- Deliberately **narrowed** `ensureBook`: a transient bad *slice* must NOT trigger a wipe-all (that
  would nuke every downloaded book) — recovery is reserved for genuine DB-image corruption.

### Resize remount fix (committed with the responsive work)
- Root cause: `ViewerPage` had **two separate** `if (isDesktop) return (...)` trees, so crossing the
  768 px breakpoint **remounted** the ~1,500-verse reader subtree.
- Unified into **one** returned tree with the reader at a **stable child position** (index 1); the
  catalog/aside render as inline `<nav>`/aside on desktop vs Mantine `<Drawer>` on mobile, plus a
  mobile aside FAB. Verified a tagged verse survives desktop→mobile→desktop without remount.

### SAH-handle race on fast reload (committed `2d4f409`) — the main fix this session
- Root cause: on a fast reload the **new** db worker called `installOpfsSAHPoolVfs` before the **old**
  worker had released its OPFS `SyncAccessHandle`s. The leader election lock (`torah-leader`) is
  **page-scoped** and releases on page unload — *before* the old worker is torn down — so it does not
  guard handle ownership.
- Fix: a **worker-scoped Web Lock `torah-sahpool`** that serializes SAH-pool ownership across worker
  generations. `worker.ts` `init` acquires it (exclusive) and holds it for the worker's lifetime
  (`await new Promise(() => {})`); the lock auto-releases only when that worker is destroyed — by
  which point its handles are freed — so the next worker's `install` waits and then acquires cleanly.
- `installPoolWithRetry` (retry on `NoModificationAllowed`) stays as a backstop for any residual gap.

### Perf investigation (no code change — measurement)
- The "really slow downloads" was **not** the crash-safety pragmas (measured ~60–100 ms) and **not**
  cache growth (download time is flat as more books are cached). It was the **boot thrash** from the
  SAH race (failed/retrying worker boots stalling the app), plus a one-time re-download after a
  corrupt-DB self-heal. Fixing the race resolved the perceived slowness.

### Verification (standalone Playwright: `scripts/repro-deploy.mjs`)
The MCP browser waits for full load between navs, masking the race — so a standalone
`chromium.launchPersistentContext` harness was added (Test A: rapid reloads counting SAH errors;
Test B: sequential downloads timing each as the cache grows). Run: `node scripts/repro-deploy.mjs [url]`.

Against the **deployed** site, after the fix went live:
- Test A: SAH errors **88 → 0** through the reload storm; app boots every time.
- Test B: downloads **2.4–3.9 s** per chumash and **flat** as the cache grows.

## Followup — the `torah-sahpool` lock could deadlock init on mobile (fixed)

Reported next: *"on my phone as a PWA the db sometimes just locks up."* The cause was the very lock
this entry added. The worker held `torah-sahpool` for its **whole lifetime** (`await new Promise(() =>
{})`), and the worker's `init` promise only resolved *after* acquiring it — and **every** DB call
awaits `init`. On mobile a PWA is **frozen, not terminated** (Page Lifecycle API), so a
backgrounded/zombie prior worker keeps holding the lock; the new worker's `init` then never resolves
and the whole app silently locks up (the shell renders, but every query spins — there is **no RPC
timeout** in `client.ts`, only a leader-change retry).

Fix (commit `a5cdb9e`): make the lock a **best-effort optimization, never a hang**. A 4 s watchdog
aborts the lock wait (`AbortController` on the lock request) and installs the VFS without the lock;
`installPoolWithRetry` still resolves the handle handoff (it may just log a couple of retries). At most
one *live* worker ever exists (the leader's), so proceeding past a stuck holder is safe. The happy-path
0-error reload handoff is unchanged.

Verified with a new repro (`scripts/repro-lock.mjs`): the page grabs `torah-sahpool` and never releases
it (stand-in for a frozen worker) — the app now boots in ~5 s (4 s watchdog + setup) instead of hanging
forever; rapid-reload regression still 0 SAH errors. Confirmed on the deployed PWA.

**Lesson:** never gate a promise that every operation awaits behind a lock held for a context's
lifetime — on mobile, "context lifetime" includes indefinite frozen states. Any such wait needs a
bounded escape hatch.

## Notes
- torah-app's deploy checks out torah-llm HEAD and rebuilds; a `workflow_dispatch` fired in the same
  breath as the push can **race the push** and check out the pre-fix commit. Confirm the deployed
  bundle actually contains the change (e.g. `grep torah-sahpool` the live `assets/worker-*.js`) before
  trusting a deployed test — the first dispatch this session deployed the stale worker and re-showed 77 errors.
- The cross-repo auto-deploy (torah-llm `Notify torah-app` → `repository_dispatch`) fires and reports
  success but does **not** reliably trigger a torah-app run (likely the default token can't dispatch
  cross-repo). In practice deploys this session were a manual `gh workflow run deploy.yml -R kolodny/torah-app`.
