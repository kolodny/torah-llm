# 024 — Complex-text recovery (the missing 18% of the corpus)

**Date:** 2026-06-10

## The gap (found by spot-check)
Comparing our Genesis 1:1 connections to Sefaria, the maintainer noticed **Abarbanel was missing**.
Investigation: **1,216 of 6,598 catalog books (18%) had NO content** — including major commentaries
**Abarbanel on Torah, Or HaChaim on Genesis, Kli Yakar on Genesis, Rav Hirsch, Ralbag, Haamek Davar
(Netziv), Radak on Genesis, Alshekh, Chatam Sofer**, …. So the earlier "entire Sefaria corpus" claim
was wrong.

**Root cause:** the Sefaria adapter only parsed *simple* texts — a nested-array `text` indexed by
`sectionNames`. Sefaria **"complex texts"** store `text` as an **object keyed by `schema.nodes`** (no
flat `sectionNames`); the adapter produced 0 refs for them and silently skipped (`if (!rows.length)
return`). The files were downloaded — they were dropped at parse time. ~869 books were complex.

## The fix
`ingest/sources/sefaria.ts` `segments()` walks the text-object and splits a complex text into
**node-qualified sub-books that match Sefaria's addressing AND the links' titles**:
- **default node** (`""`) → keeps the base title with flat numeric refs (`1:1:41`) — exactly how the
  links cite it (`Or HaChaim on Genesis 1:1:41`).
- **named nodes** → `"<title>, <node>"` sub-books, e.g. `Abarbanel on Torah, Genesis` (ref `1:1:1`),
  `Or HaChaim on Genesis, Introduction` (ref `1`). The base book for a `"…, <BaseBook>"` sub-book is
  the last node segment, so the commentary→base links derive correctly.

`sefaria-links.ts` needed **no change** — the CSV's Text column already *is* the node-qualified title
(`Abarbanel on Torah, Genesis`), so links resolve against the matching sub-book.

**Filename fix** (`shared/slice-path.ts`): some deeply-nested sub-book ids percent-encode past the
255-byte filename limit → the first slice run crashed (`SQLITE_CANTOPEN`). `sliceFileName` now falls
back to a deterministic 64-bit hash (`toc_h<hex>.sqlite`) for ids whose encoded form exceeds 200 chars
— same function in the slicer and the browser client, so they agree.

## Result (rebuilt from cached JSON — no re-download)
| | before | after |
|---|---|---|
| content books | 5,371 | **23,441** |
| segments (content rows) | 2.16M | **4.11M** |
| resolvable links | 1.97M | **3.39M** |
| boot DB (`db.sqlite`) | 2.4 MB | **7.4 MB** |
| corpus (all slices) | 3.3 GB | 7.5 GB |

**Verified in-browser:** Genesis 1:1 now lists **all 7 recovered commentaries** (Abarbanel, Or HaChaim,
Kli Yakar, Radak, Haamek Davar, Rav Hirsch, Alshekh) and they load; link resolution on the samples was
Or HaChaim **99%**, Kli Yakar **96%**. The new `publishId` means existing clients pull only the new
books **incrementally** (forward-compat, [022](022.forward-compat-storage.md)).

## Granularity tradeoff (kept, with a known cost)
Splitting at node boundaries 4×'d the book count (complex texts covering many base books → many
sub-books; ~1,929 are depth-3+, a few pathological like "a book per Hebrew letter"). I **kept full
granularity** — it matches Sefaria's addressing and is what makes the links resolve — rather than
cap depth (which would force node-in-ref like `Section, Subsection:1` and break the reader's numeric
chapter grouping/sort). The cost is the **7.4 MB boot catalog** (gzips to ~2–3 MB). If that upfront
size becomes a problem, the right fix is a **lazy/partial catalog** (ship top categories, fetch deeper
on expand) rather than corrupting refs — a future item.

## Peek "show in context" (the Tikkunei Zohar bug, same session)
A peek to `Tikkunei Zohar 17a:2` showed *only* the quoted pasuk, because that segment genuinely is
just the quote (the discussion is `17a:3`+). Sefaria's API returns the whole **section** with the
target segment marked, so `PeekPanel` now fetches the section (`getSiblings`) and renders it with the
linked segment **highlighted and scrolled into view**. Verified: the Zohar peek shows all 22 paragraphs
of daf 17a with `17a:2` highlighted. (Extends [023](023.inline-link-preview.md).)
