# 053 — Slices compressed with zstd (whole index-free corpus fits GitHub Pages' 1 GB)

**Date:** 2026-06-29

## Why

Measuring the corpus (see [the cleanup audit](052.cleanup-functions-corpus-audit.md) era): index-free slices gzip
to **1.044 GB** — just over GitHub Pages' 1 GB site limit, so the deploy could only ship a subset. Measured
codecs on a representative sample: **zstd-19 is ~24% smaller than gzip-9, brotli-11 ~27%**. A shared dictionary
gave *no* gain (big books dominate; their own window already captures the redundancy). zstd wins on
integration: a **tiny pure-JS decoder (`fzstd`, ~few KB)** with no wasm, vs brotli which needs wasm (the browser
has neither zstd nor brotli in `DecompressionStream` — verified). So zstd-19 lands the **full corpus ≈ 815 MB**,
under 1 GB, with a trivial dep — and the slices stay plain SQLite + a standard codec (rebuildable from Python).

## What shipped

- **Slicer (`ingest/slice-db.ts`)** emits `.zst` (slices + boot DB) via Node's native `zstdCompressSync`
  (level 19) instead of `gzipSync`.
- **Worker (`src/db/worker.ts`)** `streamInto` fetches `${url}.zst`, decodes with `fzstd`, then imports into the
  OPFS SAH-pool (plain-`.sqlite` fallback kept for a not-yet-re-sliced corpus). Progress is now *accurate* —
  Content-Length is the real (compressed) download size, so no more dropping `total` as with in-stream gzip.
- **PWA precache (`vite.config.ts`)** now precaches `db/db.sqlite.zst`.
- Added the `fzstd` dependency.

## Verified

- Node-`zstdCompressSync` frame ↔ `fzstd.decompress` roundtrip: byte-exact.
- `new DecompressionStream('zstd')` throws in Chromium 149 (and `'br'`) — confirms the lib is required on Pages
  (the native `Content-Encoding: zstd` path needs a server header Pages can't set).
- End-to-end in-browser (wipe OPFS → fresh boot): catalog loads from `db.sqlite.zst` (19,422 books); `ensureBook`
  downloads Genesis via the `.zst` path (564 ms) → decode → merge; Genesis 1:1 = 28 letters, 1,533 he verses.
- Subset re-slice: **19,423 `.zst` slices, boot `db.sqlite.zst` 0.91 MB** (was 1.5 MB gz), `public/db` 684 MB.
- `npx tsc` clean. (`public/db` is gitignored — CI re-slices on deploy.)

## Follow-up (open)

Slice **filenames** are still percent-encoded ids (double-encoded in the fetch URL — a cross-host footgun, and
the source of the 23k special-char names). Measured: readable transforms (camelCase/slug) **collide on the real
corpus** (2 book pairs differ only by a trailing `*`). Recommended next step: `toc_<slug>_<hash>.sqlite.zst`
(readable + auto-disambiguated by the existing 53-bit `hashId`) + a build-time uniqueness assertion. Not yet done.
